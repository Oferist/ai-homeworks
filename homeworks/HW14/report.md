# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано
- Использована база знаний по Python Data Science библиотекам.
- Чанкинг реализован фиксированным размером (60 символов, перекрытие 20).
- Использована модель `all-MiniLM-L6-v2`.
- Retrieval показал `Hit@3` = 1.00 и `MRR` = 0.95.
- Обновление базы (добавление FastAPI) позволило системе находить ответы на новые запросы.
- Mini-RAG реализован через сборку контекста из топ-k чанков и шаблонную генерацию ответа.

## 3. База знаний и подготовка документов
- Предметная область: Инструменты Python для Data Science.
- Источник документов: Локальный JSON-файл с описаниями библиотек.
- Число исходных документов: 17.
- Число чанков после разбиения: 40.
- Какой способ чанкинга использовался: Fixed-size chunking (посимвольный).
- Основные параметры чанкинга: chunk_size=60, overlap=20.
- Комментарий: Данная база знаний идеальна для retrieval, так как каждый документ тематически автономен и содержит специфическую терминологию. Это позволяет модели эмбеддингов четко разделять смысловые единицы, что критически важно для учебного построения RAG-систем.

## 4. Retrieval и индекс
### 4.1. Эмбеддинги и индекс
- Модель: `all-MiniLM-L6-v2`.
- Метрика: L2 (евклидово расстояние).
- Индекс: `IndexFlatL2`.
- Top_k: 3.
- Первые примеры: Retrieval успешно находил библиотеки по ключевым словам (напр., "array" -> NumPy).
### 4.2. Контрольные запросы и оценка retrieval
- Контрольных запросов: 10.
- Что считалось релевантным: Соответствие найденного источника ожидаемому.
- Метрики: `hit@k`, `recall@k`, `MRR@k`.
- Итоговый `hit@3`: 1.00.
- Итоговый `recall@3`: 1.00.
- Дополнительные метрики: `MRR@3` = 0.95.
- Простые запросы: "Data manipulation tool" (Pandas).
- Проблемные запросы: Запросы с общими словами типа "ML", где векторное сходство распределено между несколькими библиотеками (PyTorch, TensorFlow).

## 5. Эксперимент по параметрам retrieval и обновление базы знаний
- Параметр: `top_k`.
- Варианты: 1 vs 5.
- Изменение: Увеличение `k` до 5 ожидаемо повысило Hit Rate, но увеличило шум в контексте.
- Выбор: `k=3` выбран как оптимальный баланс точности и полноты.
- Обновление: Добавлен документ "FastAPI".
- Переиндексация: Выполнено полное пересоздание индекса через `index.add(new_embeddings)`.
- Результат: После обновления система стала корректно находить FastAPI по тематическим запросам.

## 7. Анализ
Выбранная база знаний оказалась крайне удачной для учебной задачи, так как описания библиотек контрастны и содержат четкие ключевые слова. Примененный чанкинг с параметрами chunk_size=60 и overlap=20 позволил избежать разрывов важных терминов на стыках фрагментов, что положительно отразилось на качестве поиска. Формальная оценка hit@k на 10 запросах оказалась весьма полезной, так как она наглядно подтвердила корректность работы модели эмбеддингов. Эксперимент с параметром top_k показал, что увеличение зоны поиска до 5 улучшает hit rate, но при этом сильно размывает контекст лишней информацией. Обновление базы знаний через переиндексацию индекса FAISS продемонстрировало, что добавление новых данных требует полной пересборки векторного хранилища. Наиболее показательными ошибками Mini-RAG стали случаи "семантической близости", когда модель путала библиотеку PyTorch с TensorFlow из-за схожих описаний их функционала. Также были замечены галлюцинации при запросах об инструментах, отсутствующих в базе, когда система выдавала ответ по наиболее похожему по вектору фрагменту. Основным ограничением текущего решения является отсутствие механизма reranking, который мог бы отсеять шумные результаты поиска. Учебный конвейер наглядно демонстрирует, что качество итогового ответа в RAG-системе критически зависит от точности попадания в релевантный контекст на этапе retrieval. Данный проект подтвердил, что даже простая комбинация эмбеддингов и FAISS позволяет собрать рабочее поисковое решение.

## 8. Итоговый вывод
- Базовый конвейер: SentenceTransformer + IndexFlatL2 + простой RAG-шаблон.
- Quality Retrieval: Качество поиска напрямую зависит от семантической "чистоты" чанков.
- Связь RAG: RAG не может исправить плохой retrieval; качество контекста — залог успеха всей системы.
